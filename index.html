Summary

“Koko” is a bot created for the instant messaging platform, Discord that is currently used in dozens of Discord servers. Koko is designed to be a companion bot for communities built around sites on the Wikifarm, Wikidot, and it allows users to search for pages on Wikidot sites by name or by author, as well as be notified of any new or deleted pages. To do this, I use the Discord.NET API in C# to operate the bot using asynchronous programming, and I use third-party scraping libraries to look for new pages and keep track of their authors and ratings.

Koko was created in order to give users a more convenient way to use Wikidot sites. Despite how widely used Wikidot is, many of its features are no longer maintained, including its search feature. Because of that, Koko greatly increases the quality of life for those using Wikidot. Additionally, Koko’s feature of notifying users of new pages allows moderators of Wikidot sites to more quickly be aware of new content, which makes it easier to moderate.

Koko was originally created on my own. However, as the scope of the project grew larger, I created a team of two other people to work collaboratively on maintaining and updating the bot with new features.

Analysis

This project was my first time developing code as a team with other people, so through it, I was able to learn a lot about how to proofread other people’s code, and coordinate a team towards a goal, while making sure to take advantage of the efficiency that comes with division of work, while also keeping enough coordination to make sure each team member is producing good code that works with each other.

I also learned a lot about using APIs and third party scraping. Although the .NET version of the Discord API is not nearly as well documented as some other commonly used versions, I was able to figure out how to use it with some research. Wikidot itself also has its own API, but it was behind a paywall and hasn’t been maintained for over a decade, so I opted to use third party scraping for that aspect of the bot. While scraping wasn’t ideal, it taught me how to problem solve and how to come up with clever ways to get the information I needed.

Since Koko is used in dozens of Discord servers, some of which have thousands of members, I was able to get the experience of having my code used by a large number of people in a way I hadn’t before. This was an interesting experience because it showed me how valuable it can be to speak directly to a user base and determine which features they would find helpful in future updates, as well as showing me how important it is to rigorously test your code when something you may have missed could be found by a user base in the thousands in only a few days.

Summary

This project was a python program that implemented my own personal algorithm for calculating the results of single transferable vote elections. Single transferable vote elections are an alternative of the widely-used first past the post election design, in which it has voters submit an ordered list of their ideal candidates rather than a single candidate, and a winner is calculated from those lists in a way that doesn’t punish voters for putting a less popular candidate as their first choice. These types of elections work very well when used on a large scale; however, I found that the most widely used algorithms for calculating the results of a single transferable vote election break down when used with a very small number of voters, such as decisions voted on by a small elected council. To fix this problem, I designed a new algorithm for calculating the results of single transferable vote elections and developed a program in Python that implemented this algorithm.

Analysis

This project taught me a lot about problem solving and algorithmic thinking. The end goal wasn’t very concretely defined since part of the problem that the algorithm was aiming to solve was the difficulty in quantifiably measuring its own results. Due to this issue of actually measuring the performance of the algorithm, I had to shift my focus from finding the “best” algorithm if such a thing exists to instead creating an algorithm that fixes the clear issues in the widely-used algorithm, making it, at the very least, strictly better than that algorithm. This was accomplished by keeping the basic parts of the widely used algorithm for single transferable vote elections and changing aspects of the structure in intuitive ways to fix the blindspots present. Through this I was able to learn new ways of looking at finding the solution to a given problem.

Implementing the algorithm in Python also taught me a lot. Although I intentionally made the algorithm simple and intuitive in concept, that simplicity did not necessarily transfer to the actual implementation, especially while keeping the time complexity of the algorithm as low as possible. To accomplish this, I went through a lot of iterative steps to accomplish this. I started by implementing a brute force program that essentially amounted to throwing a thousand weighted dice that each contained a simpler version of the algorithm, that together would average out to the correct result of the full algorithm. This approach got the right answer, but with a dismal time complexity. From here, I did research on data structures and was able to design a smarter implementation that used trees to cut down the time complexity significantly. Through this, I learned a lot about problem solving and how to think of a problem in different ways to find the most efficient solution.

Summary

“GTFSTool” was a 10-week team project I undertook as part of my Software Engineering Tools & Practices class. It entailed creating an application in Java/JavaFX, which allowed users to input CSV files that use the Google Transit Feed Specification that define the information relating to the bus route system of a given city. From the information in these files, the generates an internal map of the relationships between hundreds of bus routes, stops, and individual trips and displays various information about the bus system to the user including a map that displays all stop locations, and routes along with the frequency that a bus will go along any given route. This project was implemented using test-driven-development and acceptance criteria, and it made use of the model-view-controller and observer design patterns with a focus on maintainability and the reduction of technical debt.

Analysis

This project was one of the largest projects I’ve worked on and was made more complex by the fact that I was working with a team of three of my peers. Even though we were all technically equals, I found myself taking on a managerial role and personally and had to evaluate the strengths and weaknesses of my teammates to determine how the workload of the project specifications could be most effectively split up between the four of us. Through this I learned a lot about how to organize a team of coders of different skill levels in an effective way.

One thing I earned from this project was a new-found appreciation for test-driven-development. What had before seemed like simply a necessary tedium became essential when it came to making sure that all four members of the team were designing their code in a way adhered to the projected specifications and kept the technical debt low. Through this process, we all worked together to develop a suite of tests with nearly 100% coverage, which made code review and bug fixing much easier.

Our project was based on displaying complexly related data to the user in multiple views in a graphical user interface with framework that allowed the user to make changes to the data in the system, so when we were creating our initial design, it was clear that model-view-controller and observer design patterns would be the best design patterns for making the project as easy as possible for four people to work on it simultaneously.
